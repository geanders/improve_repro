3.9 Knitted documents

If you have coded using a scripting language like R or Python, you likely have already seen many examples of knitted documents. For both these languages, there are many tutorials available that are created as knitted documents. Figure @ref{fig:xcmsexample} shows an example from the start of a vignette for the xcms package in R. This is a package that helps with pre-processing and analyzing data from LC-MS experiments. You can see that this document includes text to explain the package and also example code and the output from that code.

Figure 3.1: An example of a knitted document. This shows a section of the online vignette for the xcms package from Bioconductor. The two types of content are highlighted: formatted text for humans to read, and executable computer code.

In this module we will describe the characteristics of knitted documents and how the technology behind them works. We will then talk about how you can create them specifically using R markdown.

The defining characteristic of a knitted document is that it interweaves two elements: first, executable code, and, second, formatted documentation that is meant for humans to read. In the figure with the example from the xcms vignette, we have highlighted the areas that demonstrate these two pieces (Figure [x]). Later in this module we will describe how a knitted document can incorporate these two elements. First we want to highlight why you might want to use knitted documents to document your own research code, especially for pre-processing protocols.

There are several advantages to using knitted documents when writing code to pre-process or analyze research data. First when you have written your code inside a knitted document the code itself is checked every time you render the document. In other words you are checking your code to ensure it operates as you like as you build your document. This helps to increase the reliability of the code that you have written, especially as you work to reapply it to new data set. As you work with open source code, you will find that the packages can change some overtime, as the developers continue to develop them. By including code that is checked each time of document is rendered, you can quickly identify when a change an open source software effects the analysis that you were conducting or the pre-processing and work 2 adapt to any changes quickly.

Second, when you write a pre-processing protocol or other document that includes executable code allows you to easily rerun the code as you update the data set or adopt the code to work with a new data set. Another approach would be to run all your code and copy the results into a document in a word processing program like word or Google Docs. If you did that, you would need to recopy all results every time you shipped the code or add new data. By contrast, when you are working with a knitted document the rendering process don’t executes the code and incorporates the results into a nicely formatted final document. The use of knitted documents therefore can substantially improve the efficiency of pre-processing and analyzing your data and generating the reports that summarize this process.

3rd documents that are created in knitted format are created using plain text. Plain text files can easily be tracked well and clearly using Version Control tools like git. This substantially increases the transparency of the process. It allows you to clearly document changes you make in the document, step by step. You can document who made the change, and that person can include a message about why they made the change. The full history of changes is documented and can be searched to explore how the document evolved and why.

The final advantage of using knitted documents, especially for pre-process research data, is that it allows the code to be clearly and thoroughly documented. This can help increase the reproducibility of the process. In other words it can help ensure that another researcher could repeat the process that you did, making adaptations as appropriate for different data sets. It also means that you can remember exactly what you did and reuse are adopt the code as you work with other data sets. An alternative way of creating code to pre-process data would be through a code script. A code script does allow you to include documentation about the code through code comments. However these are much less expressive and harder to read the nicely formatted text. A knitted document allows you to write the documentation in a format that is clear and attractive for humans to read, well including code that is clear an easy for a computer to execute.

Next We will describe how needed documents work. there are seven components for how did it documents work. We have listed them here and in the following paragraphs will describe each more fully:

  1. Knitted documents start as plain text
  2. A special section at the start of the document (preamble) gives some overall directions about the document
  3. Special combinations of characters indicate where the executable code starts
  4. Other special combinations show where the regular text starts (and the executable code section ends)
  5. Formatting for the rest of the document is specified with a markup language
  6. You create the final document by rendering the plain text document. This process runs through two software programs.
  7. The final document is attractive and read-only—you should never make edits to this output, only to your initial plain text document.

The next characteristic of knitted documents is that formatting for the rest of the document, other than executable code, is specified using what is called a markup language. When you were writing in plain text, you do not have buttons to click on for formatting like bold, italic, font size, and so on. Instead you use special characters or character combinations to specify formatting in the final document. For example in marked down, you’ll see around a word or phrase into asterisks to make it bold. TTo write “this” in the final document, you’ll write "**this**" in the plain text initial document. For example when you get to a word that you want in italics, you essentially need to tell the computer start italics, then say the word, then say to end the italics. You do this using the special formatting characters that are specified by the markup language that you use. The final characteristics of knitted documents is that to create the final document you will render the plain text document and that this will create an attractive document that is read-only. When you render the document it will be run through to software program the first will look only four sections with executable code. It will execute that code and take any output, including numeric results, other data results, figures, and tables, and insert those at the relevant spot in the document. Next these output files will be run through another software program that will look for all the formatting instruction and render the final document in an attractive format for humans to read. this output can be in a number of file format, depending on your specifications, including a PDF document, an HTML file, or a Word document. You should consider the final document, regardless of the output format, as read-only. This means that you should never make edits or changes to the final version of the document. Instead you should make any changes to your initial plain text file. this is because the rendering process will overwrite any previous versions of the final document. Therefore any changes that you have made to your final document will be overwritten anytime you re-render from the initial plain text document.
